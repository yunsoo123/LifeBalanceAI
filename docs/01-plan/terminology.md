## Core Concepts

Definitions for key terms used across the product, design, and codebase. Each term is explained in plain language so anyone can understand it without prior context.

---

### User-Facing Terms

1. **Inbox** — The blank page where users type or speak their goals, tasks, and thoughts freely without structure. It is the main capture surface before anything is organized into goals or calendar.

2. **Brain Dump** — The act of writing or speaking thoughts freely into the Inbox without worrying about organization. The goal is to get ideas out quickly; structuring happens later with AI or manual steps.

3. **AI Schedule** — A realistic weekly time allocation generated by AI based on the user’s goals, hours per week, and constraints. It respects total available time and suggests a feasible plan.

4. **Smart Calendar** — A calendar that auto-places activities based on optimal times and avoids conflicts. It uses user preferences (e.g. morning vs evening) and energy levels to suggest when to do what.

5. **Notion-style Notes** — Rich-text notes linked to calendar events, supporting code blocks, checklists, headings, and formatting. They behave like a lightweight wiki attached to events.

6. **Weekly Review** — An AI-generated Sunday report showing performance, completed hours per goal, and improvement suggestions. It helps users see what was done and what to adjust next week.

7. **Goal** — A high-level activity the user wants to pursue over time (e.g. “App Development”, “Poker Grinding”). Goals have target hours per week, priority, and optimal time of day.

8. **Event** — A specific time block on the calendar (e.g. “App Dev 2–5pm Monday”). Events can be linked to a goal and to a generated schedule.

9. **Reality Check** — AI validation that total planned hours do not exceed 168 hours per week and that sleep and rest are accounted for. It prevents unrealistic schedules.

10. **Energy Level** — The user’s self-reported energy state (e.g. high / medium / low) used to suggest when to schedule demanding vs lighter tasks.

---

### Technical Terms

11. **PDCA** — Plan–Do–Check–Act: a cycle for feature development. We plan (scope, success criteria), do (implement), check (gap analysis vs design), and act (fix gaps, then report).

12. **Match Rate** — The percentage of design requirements that were successfully implemented in code. We aim for ≥90% before considering a feature complete.

13. **Gap Analysis** — A comparison between the design document and the actual implementation. It lists missing or differing items so we can fix them systematically.

14. **Design Token** — A reusable design value (color, spacing, typography, radius) defined in one place (e.g. `lib/design-system.ts`). Components use tokens instead of hardcoded values for consistency.

15. **RLS (Row Level Security)** — A Supabase/PostgreSQL feature that restricts which rows a user can read or write. In our app, users only see and edit their own data.

16. **Zod Schema** — A TypeScript library for runtime validation. We use it to validate AI responses, API inputs, and form data so invalid data is caught before use.

17. **Offline Queue** — A mechanism that stores user actions (e.g. create event, add note) when the device is offline and syncs them to the server when the connection is restored.

18. **Rate Limiting** — A cap on how often a user can call the AI API (e.g. 10 requests per minute on Free, 50 on Pro). It protects cost and availability.

19. **Subscription Tier** — The user’s plan level (Free, Pro, Teams). It determines feature access, AI limits, and billing.

20. **AI Token** — The unit of usage for the OpenAI API (input and output text). We track tokens per call for cost monitoring and budgeting.

---

### Feature-Specific Terms

21. **Conversational Scheduling** — A flow where the AI asks clarifying questions (e.g. hours, preferences, constraints) before generating a schedule, instead of generating from a single prompt.

22. **Auto-Placement** — An algorithm that assigns events to calendar slots based on optimal time, duration, and conflict avoidance, reducing manual drag-and-drop.

23. **Gesture Control** — Mobile interactions such as swipe-to-delete, pull-to-refresh, and pan gestures. They make the app faster to use on touch devices.

24. **Touch Target** — The minimum tappable size for interactive elements. We use ≥44×44 dp on iOS and ≥48×48 dp on Android so taps are easy and accurate.

25. **Empty State** — The UI shown when a list or collection has no items (e.g. no goals, no events). It explains what the section is for and shows a clear next action (e.g. “Add your first goal”).

## Conventions

Naming rules used in the codebase and docs so names are predictable and consistent.

| Context | Convention | Example |
|--------|-------------|---------|
| Components, types, interfaces | **PascalCase** | `CalendarGrid`, `Schedule` |
| Functions, variables | **camelCase** | `generateSchedule`, `userId` |
| Constants | **UPPER_SNAKE_CASE** | `MAX_GOALS`, `AI_MODEL` |
| Files, folders | **kebab-case** | `ai-chat.tsx`, `use-schedule.ts` |
| Database tables, columns | **snake_case** | `user_id`, `created_at` |
