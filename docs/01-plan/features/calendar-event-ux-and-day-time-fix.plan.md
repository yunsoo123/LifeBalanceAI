## 1. 요청 해석 (표면 vs 실제 의도)

| 표면 요청 | 실제 의도 |
|-----------|------------|
| "캘린더에서 일정 만들기가 거의 불가능 (UI/UX 디자인 문제)" | **New Event 모달**이 기기/환경에서 겹쳐 보이거나 가독성이 떨어져, 실제로 입력·완료가 어렵다. 단순 꾸미기가 아니라 **사용 가능한 폼**으로 만들어야 함. |
| "금요일 오전 일정 만들어 달라고 했는데 월요일로 추가됨" | **Inbox/Schedule에서 사용자 의도(요일·시간)**가 DB에 반영될 때 잘못 매핑됨. "금요일 오전" → Friday 9:00 이어야 하는데 Monday로 들어감. |
| "달력에만 추가되고 타임테이블엔 없음" | 같은 **events** 테이블을 쓰므로 이론상 둘 다 나와야 함. **요일/시간대 오류**로 다른 주에 들어갔거나, **주간 범위/타임존** 계산 차이로 타임테이블 주에 안 잡힐 수 있음. |
| "기능상 문제가 많고 UI/UX는 블럭 형태, 요즘 앱 트렌드" | 일정 생성·편집을 **카드/블록 단위**로 보여주고, 한 화면에서 요일·시간을 직관적으로 고르는 구조로 바꾸고 싶다는 의미. (예: 타임테이블 셀 탭으로 추가, 블록 드래그 등) |

---

## 2. 전제 조건

- **P1**: 캘린더 "Add Event" 모달은 현재 ScrollView + 라벨 + TextInput 구조이나, 특정 기기/해상도/키보드에서 **레이아웃이 깨지거나 겹쳐 보이는 문제**가 있음 (이미지에서 확인).
- **P2**: Schedule에서 "첫 번째 활동"의 기본 요일이 **index % 7 → 0 (월요일)** 로 고정되어 있어, 사용자가 "금요일 오전만"이라고 해도 **초기 슬롯이 월요일**로 나올 수 있음. AI가 요일을 명시해도 Schedule 쪽에서 그걸 반영하지 않으면 월요일로 노출·캘린더 등록됨.
- **P3**: 타임테이블과 달력은 **동일 events 테이블**을 사용하므로, "달력에만 있고 타임테이블엔 없다"는 현상은 **저장된 요일/시간이 타임테이블이 보는 주와 다르거나**, 타임존/주 시작일 계산 차이일 가능성이 큼.
- **P4**: "블럭 형태"는 **모던 캘린더 앱처럼** 시간축 블록·카드형 입력·드래그 등으로 정의할 수 있음. 범위가 크므로 **우선순위**를 둬야 함.

## 3. 알 것 / 모를 것

| 구분 | 내용 |
|------|------|
| **알 것** | (1) Add Event 모달 코드: ScrollView + 고정 라벨/필드. (2) Schedule의 getEffectiveSlot: override 없으면 dayOfWeek = index % 7 → 첫 활동 = 월요일. (3) Inbox→캘린더 등록 시 placeParsedEvents는 days[]와 startTime 기반; AI가 "금요일"을 days: [4]로 주면 배치는 금요일로 됨. (4) Schedule 대화형 플로우에서 "금요일 오전만"이라고 했을 때 **AI 응답에 요일이 포함되는지**, 그리고 **Schedule UI가 그 요일을 초기 슬롯으로 쓰는지**는 코드만으로는 일부 추론 가능. |
| **모를 것** | (1) **정확한 재현 환경**(OS, 해상도, 키보드 열림 여부)에서 모달 겹침이 나는 조건. (2) 스크린샷의 "축구 월·9:00"이 **AI가 월요일로 생성한 것인지**, **Schedule 기본 슬롯(첫 활동=월) 때문인지**. (3) "타임테이블에 없다"가 **다른 주에 저장돼서** 안 보이는 것인지, **같은 주인데 쿼리/표시 버그**인지. |

## 4. 접근 (2가지 이상, 트레이드오프)

### 접근 A: 모달 UX만 우선 수정 + 요일/시간 버그만 수정

- **내용**:  
  - New Event 모달: 겹침 방지(고정 높이/스크롤/블록형 필드), i18n, 날짜·시간 선택기 명확화.  
  - Schedule: 사용자/AI가 지정한 "금요일 오전"을 반영하도록, 대화 결과 또는 파싱 결과에서 **요일·시간**을 추출해 **초기 슬롯(dayOfWeek, startMinutes)** 에 넣고, "캘린더에 등록" 시 그 값으로 insert.  
  - 타임테이블: 이벤트 로드 시 **주 범위(월요일 00:00 ~ 일요일 23:59, 로컬)** 와 **start_time** 비교가 일치하도록 보정(필요 시 타임존/주 시작일 통일).
- **장점**: 범위가 제한돼 빠르게 배포 가능.  
- **단점**: "블럭 형태"의 큰 UX 변경은 하지 않음.

### 접근 B: 캘린더 일정 생성/편집을 블록·타임테이블 중심으로 재설계

- **내용**:  
  - "Add Event"를 모달 대신 **타임테이블 셀 탭 → 해당 요일/시간에 블록 추가** 또는 **주간 뷰에서 빈 블록 탭**으로 만드는 플로우로 전환.  
  - 기존 모달은 "상세 편집(제목, 설명 등)"만 담당하거나 제거.  
  - Schedule/Inbox에서 "캘린더에 등록" 시 요일·시간이 AI/사용자 의도와 일치하도록 위와 동일하게 수정.
- **장점**: "요즘 앱 트렌드", 블럭 형태에 부합.  
- **단점**: 설계·구현 규모가 크고, 기존 모달 의존 사용자에게는 플로우 변경이 필요.

### 접근 C: A 먼저, B는 Design으로만 정리 후 단계적 적용

- **내용**:  
  - **1단계**: 접근 A(모달 안정화 + 요일/시간 버그 + 타임테이블 주/표시 일치)를 Plan/Design/Do로 진행.  
  - **2단계**: "캘린더 블록형 생성/편집"은 별도 feature로 Design 문서만 작성해, 이후 스프린트에서 타임테이블 탭→블록 추가 등 구현.
- **장점**: 당장 불가능한 일정 생성은 해소하고, 블럭 UX는 로드맵으로 명확히 함.  
- **단점**: 블럭형은 당장 코드로 나오지 않음.

---

## 5. 권장

- **권장**: **접근 C**.  
  - 이유: (1) "거의 불가능할 지경"을 먼저 해소하는 것이 우선. (2) 금요일→월요일, 타임테이블 누락은 **기능 버그**이므로 데이터/로직 수정이 선행되어야 함. (3) 블럭 형태는 Design으로 요구사항을 고정한 뒤, 모달 수정이 안정된 다음에 단계적으로 도입하는 편이 리스크가 적음.

---

## 6. 이 Plan에서 다룰 범위 (접근 C 1단계)

- **In scope**  
  - **캘린더 New Event 모달**: 겹침/가독성 해결(블록형 라벨·필드, 스크롤·키보드 대응), i18n, 날짜/시간 입력 명확화(필요 시 date/time picker 또는 명시적 HH:MM 검증).  
  - **Schedule → 요일/시간 반영**: 사용자/AI가 "금요일 오전" 등으로 지정한 경우, 그에 맞는 **dayOfWeek(4), startMinutes(9*60)** 를 초기 슬롯 및 "캘린더에 등록" 시 사용. (Schedule generator/API가 요일을 내려주면 그대로 사용; 안 주면 기존 기본값 유지.)  
  - **타임테이블 표시 일치**: 이벤트가 저장된 주와 타임테이블 주가 같을 때 반드시 표시되도록, 주 범위·타임존 계산 점검 및 필요 시 수정.  
- **Out of scope (별도 feature)**  
  - 타임테이블 셀 탭으로 "블록 추가"하는 새 플로우, 드래그로 시간 변경 등 블록형 대규모 리디자인.

## 7. 인수 조건 (1단계 완료 기준)

- [ ] New Event 모달이 대상 기기에서 **겹치지 않고** 제목/설명/날짜/시작시간을 명확히 입력·저장할 수 있다.

- [ ] 해당 주에 저장된 이벤트는 **달력(월별)과 타임테이블(주간)** 모두에서 보인다.
