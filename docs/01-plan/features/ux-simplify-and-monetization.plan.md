## 1. 요청 해석 (Deep-think)

| 표면 의견 | 실제 의도 | 해석 |
|-----------|-----------|------|
| 기능이 너무 복잡하고 중복된다 | Inbox·Schedule·Calendar·Notes 등 진입점과 액션이 여러 갈래로 흩어져 있어 **한 가지 목표(예: “오늘 할 일 넣기”)를 이루는 경로가 불명확**함. 버튼/용어 중복(예: 구조화 vs 자동 일정)으로 인지 부하. | **주요 시나리오별 단일 경로** 정리, **중복 액션 통합** 필요. |
| 일정 생성 후 몇 시~몇 시인지 확인하기 어렵다 | 이벤트 카드에서 **시작·종료 시간이 작거나 한 줄에만** 있어서 스캔이 어렵다. “언제인지”가 곧바로 눈에 안 들어옴. | **시간 정보를 시각적으로 강조**(타임칩, 시간 전용 라인, 또는 카드 상단 시간 뱃지). |
| Inbox에서 구조화해도 바로 일정 생성이 안 된다 | 구조화(파싱) 후 **한 번 더 “자동 일정”을 눌러야** 캘린더에 들어간다고 인지. “구조화 = 일정 반영”으로 기대하는데 실제는 2단계. | **구조화 → 캘린더 반영**을 **한 번에** 하거나, **단일 CTA**(“캘린더에 추가”)로 명확히. |
| 타임테이블 찾기가 어렵다 | 타임테이블이 **Calendar 탭 안의 Month/Timetable 토글**에 숨어 있어, “주간 시간표”를 찾는 사용자가 **Calendar만으로는 발견하기 어렵다**. | **타임테이블 노출 강화**: 진입점(탭/헤더/온보딩) 또는 첫 진입 시 뷰 기본값. |
| UI/UX가 목업 수준, 메모장 같다 | 카드·여백·색상이 있지만 **정보가 “글자 나열”**로 느껴져, **위계·시각적 구분·프리미엄감**이 부족. **돈 내고 쓰고 싶은 수준**까지 갈 여지가 있음. | **정보 블록화**(시간/제목/메타 분리), **일관된 카드·타이포**, **빈 상태·성공 피드백** 정교화. |

## 2. 전제 조건

- **P1**: 현재 개발 단계에서 **대규모 기능 삭제보다는 “경로 단순화 + 노출 강화 + 시각 정교화”**가 목표.
- **P2**: 수익화(지갑 연다)를 위해서는 **핵심 가치가 3초 안에 이해**되고, **주요 시나리오가 2~3탭/2~3탭 안에 완결**되어야 함.
- **P3**: Inbox → (구조화) → Calendar 흐름과 Schedule → (생성) → Calendar 흐름이 **겹쳐 보이지 않도록** 역할을 나누거나, **하나로 합치는 옵션**을 검토할 수 있음.
- **P4**: 타임테이블은 **같은 events 데이터**를 주간 그리드로 보는 뷰이므로, “찾기 어려움”은 **발견성(discoverability)** 문제.

## 4. 접근 (2가지)

**A. 최소 변경 (Quick wins)**  
- **시간 가시성**: 이벤트 카드에서 시작·종료 시간을 **한 줄에 크게**(예: `text-base font-semibold`) 또는 **시간 전용 뱃지/칩**으로 표시.  
- **Inbox 직행**: “일정으로 구조화” 성공 시 **같은 블록 안에 “캘린더에 N개 추가”** 한 버튼만 두고, “구조화”와 “자동 일정” 라벨/역할 통합(한 번에 파싱+배치 옵션 또는 1탭 안에서 완결).  
- **타임테이블 발견**: Calendar 진입 시 **기본 뷰를 Timetable**로 하거나, **헤더에 “주간 보기”** 등으로 노출.  
- **UI**: 이벤트 카드에 **시간·제목·설명** 블록 구분, 카드 패딩·보더 정리.

**B. 시나리오 재정의**  
- A에 더해, **“할 일 넣기” 시나리오 1개**를 정해 Inbox vs Schedule 역할을 명확히(예: Inbox = 빠른 덤프+한 번에 캘린더, Schedule = 대화형 주간 설계).  
- 타임테이블을 **탭 레벨**로 올리거나(Calendar → Week/Month 분리), 온보딩/툴팁으로 “주간 시간표는 여기” 안내.  
- 유료 기능(Pro) 후보를 정하고, 해당 화면을 **프리미엄 느낌**으로 다듬기.

**권장**: **A 먼저** 적용해 피드백 5가지를 해소한 뒤, 지표·피드백 보고 B(시나리오·수익화)를 별도 PDCA로 진행.

## 5. 인수 조건 (Acceptance Criteria)

- [ ] **시간 가시성**: 일정(캘린더·스케줄 결과)에서 **“몇 시 ~ 몇 시”**가 **한눈에** 보인다(큰 글자 또는 시간 전용 블록/뱃지).  
- [ ] **Inbox → 일정**: 구조화(파싱) 후 **한 번의 명확한 액션**(“캘린더에 추가” 등)으로 일정 생성 가능. 중복 버튼/용어 정리.  
- [ ] **타임테이블 발견**: Calendar 탭에서 **타임테이블(주간 보기)**를 **바로 찾을 수 있음**(기본 뷰 또는 헤더/토글 강조).  
- [ ] **복잡도/중복 감소**: Inbox·Schedule 각각 **주요 CTA 1~2개**로 정리, “뭘 눌러야 할지” 혼동 최소화.  
- [ ] **UI 정교화**: 일정·리스트가 **“글자 나열”이 아닌 블록/카드 위계**로 보이도록(시간·제목·메타 분리, 여백·타이포 일관).

## 6. 범위 (Scope)

**In scope**  
- 이벤트 카드: 시작·종료 시간 시각 강조(캘린더, 필요 시 스케줄 결과).  
- Inbox: 구조화 후 **단일 CTA** 또는 **파싱+배치 한 번에** 옵션, 버튼 라벨/역할 정리.  
- Calendar: 타임테이블 **기본 뷰** 또는 **헤더 “주간”** 노출 강화.  
- 공통: 카드·타이포·여백 정리(design-system·v0 컨셉 정렬).

**Out of scope (현재 Plan)**  
- 탭 개수/구조 변경(타임테이블 전용 탭 등) — B에서 검토.  
- 유료 기능 스펙 및 가격 — 별도 기획.

## 8. 개선책 요약 (지갑 여는 앱으로 가기 위한 단계)

**현재 개발 단계에서 할 수 있는 것**

1. **시간 가시성**  
   - 일정 카드에 “9:00 AM – 11:00 AM” 같은 **시간 전용 라인**을 제목 옆/위에 크게 배치.  
   - 스케줄 결과·활동 리스트에도 “요일 · 시작시간”을 같은 패턴으로 노출.

2. **Inbox “바로 일정 생성”**  
   - **옵션 A**: “일정으로 구조화” 클릭 시 **파싱 + 자동 배치를 한 번에** 수행(구조화 → 즉시 캘린더 추가).  
   - **옵션 B**: 파싱 후 **버튼 하나**만 두기 — “캘린더에 N개 추가” (기존 “자동 일정”과 동일 동작, 라벨만 명확히).  
   - “구조화하기”와 “자동 일정”이 **같은 영역에 두 개** 있는 현재 구조를 정리해, **한 흐름 한 CTA**로 정리.

3. **타임테이블 찾기**  
   - Calendar 탭 **기본 뷰를 Timetable(주간)** 으로 두거나,  
   - Month 위에 **“이번 주” / “주간 보기”** 링크를 두어 타임테이블로 한 번에 이동.

4. **복잡도/중복 감소**  
   - Inbox: **입력 → (선택) 구조화 → “캘린더에 추가”** 한 경로로 통일.  
   - Schedule: **대화 → 일정 생성 → “캘린더에 등록”** 유지하되, 결과 카드에서 **시간 정보**를 더 크게 표시.

5. **UI 정교화 (메모장 탈피)**  
   - 이벤트·항목을 **카드 단위**로 묶고, **시간 / 제목 / 설명**을 시각적으로 블록 분리.  
   - `docs/design/v0-pages-and-concept.md`의 카드·타이포·여백을 적용해 **토스·노션 수준**으로 위계 정리.  
   - 빈 상태·성공 토스트/알림 메시지를 짧고 명확하게.

**수익화(지갑)를 위한 다음 단계 (별도 기획)**  
- **Pro 기능 후보** 정하기(예: 무제한 AI 일정 생성, 주간 리뷰 인사이트, 노트 검색).  
- 해당 기능이 있는 화면을 **프리미엄 느낌**으로 다듬기.  
- 온보딩에서 **핵심 가치**(“한 줄로 일정까지”)를 3초 안에 전달.
