## 0. 에이전트 규칙에 비판적 사고 추가

### 표면 요청
"에이전트 규칙에 비판적 사고를 추가해서 내가 말하는 걸 비판적 사고로 생각할 수 있도록 하길 바람."

### 해석
- **의도**: 사용자 발언을 "그대로 실행"이 아니라 **가정 검증·대안·트레이드오프**를 거쳐 해석하도록 규칙을 보강.
- **비판적 사고** 여기서: (1) 사용자 말의 전제가 맞는지, (2) 말하지 않은 대안은 없는지, (3) 구현 시 부작용은 없는지 묻고 정리하는 절차.

### 제안 규칙 문안 (추가할 내용)

```markdown
### 비판적 사고 (Critical Thinking)
- 사용자 요청을 받으면 먼저 **전제·범위·대안**을 짧게 정리한 뒤 구현 또는 제안하라.
- "A 해줘" → (1) A가 맞는 해결인지, (2) A 말고 B/C는 없는지, (3) A의 부작용(다른 탭/기능 영향)은 없는지 확인한다.
- 모호한 표현("이쁘게", "편하게")은 **구체 스펙 1~2안**으로 바꿔 제시하고, 사용자 선택을 받은 뒤 진행한다.
- "지금이 더 낫다면 지금 상태로 냅둬도 됨" 같은 조건부 요청은, **비교 기준(사용자 가치·유지보수)**을 명시하고 "유지 권장" vs "변경 권장"을 이유와 함께 답한다.
```

- **위치 제안**: `.cursorrules`의 "Core Principles" 다음 섹션으로 넣거나, `.cursor/rules/critical-thinking.mdc` 같은 별도 규칙 파일로 두고 프로젝트에서 참조.

## 1. 캘린더 일정 추가 시 폰트 겹침 → 새 페이지 vs 큰 박스(포스트잇)

### 표면 요청
"캘린더 탭에서 일정 추가하려고 하면 폰트가 겹쳐서 어려움. 아예 새로운 페이지를 빼거나 큰 박스가 튀어나오게 해서(A4 위에 포스트잇 붙이는 느낌으로) 작성 편리하게 만들어야 할 듯."

### 비판적 정리
- **전제**: 현재도 모달 내부는 블록형·margin·minHeight로 수정했으나, **특정 기기/해상도/키보드**에서 여전히 겹침이 발생한다는 가정.
- **가능 원인**: (1) Modal + KeyboardAvoidingView + ScrollView 조합에서 뷰가 압축되며 텍스트 레이어가 겹침, (2) NativeWind 클래스와 style 혼용 시 특정 환경에서 적용 순서/특이성 문제, (3) 작은 화면에서 `maxHeight: '85%'` 안쪽이 더 줄어들어 여백 부족.
- **"새 페이지" vs "큰 박스(포스트잇)"**는 **진입 방식**과 **시각 메타포** 차이.

### 접근 A: 새 페이지 (전용 라우트)
- **내용**: "일정 추가" 시 모달이 아니라 `/calendar/new` 또는 `/(tabs)/calendar/new` 같은 **전용 화면**으로 이동. 제목/설명/날짜/시간을 풀스크린 레이아웃으로 배치.
- **장점**: 오버레이/키보드/겹침 이슈를 모달에서 분리해 제거 가능. 뒤로가기로 취소 명확.
- **단점**: 탭 컨텍스트(캘린더)를 벗어남. "빠르게 하나 더 추가" 플로우는 한 단계 늘어남.

### 접근 B: 큰 박스(포스트잇 on A4) — 모달 개선
- **내용**: 모달을 **전체 화면에 가깝게** 두지 말고, **고정 비율의 카드**(예: 세로로 긴 A5~A4 비율) 하나가 화면 중앙에 떠 있고, 배경은 반투명 어둡게. 카드 자체는 흰색/연한 회색 + 부드러운 그림자로 "종이 위 포스트잇" 느낌. 내부는 **단일 스크롤 영역**, 필드 간 여백을 크게(예: 20~24px), 폰트 크기·라인 높이를 명시해 겹침 여지 제거.
- **장점**: 캘린더 탭을 벗어나지 않음. "포스트잇" 메타포로 작성 영역이 한 덩어리로 인지됨.
- **단점**: 여전히 모달이므로 기기별 키보드/ SafeArea 차이는 대응 필요. 매우 작은 화면에서는 카드 비율 조정 필요.

### 권장
- **1단계**: **접근 B(큰 박스)** 먼저 적용. 이유: (1) 캘린더 맥락 유지, (2) 변경 범위가 모달 하나로 제한되어 리스크 작음, (3) "포스트잇" 스타일은 디자인 시스템(카드·그림자)과 맞음.  
- **구체 스펙**: 모달 콘텐츠 래퍼를 `maxWidth`(예: 400), `minHeight`(예: 520), `aspectRatio` 또는 고정 높이로 두고, 배경은 `bg-black/50`, 카드 배경은 `bg-white` + `shadow-2xl` + `rounded-2xl`. 내부 padding 24px, 필드 간 20px. 라벨 `text-[13px]`·입력 `text-[16px]` 고정, `lineHeight` 명시.  
- **재현 후에도 겹침이 남으면** 접근 A(전용 페이지)를 설계·도입.

## 2. Inbox vs Schedule 중복·통합 검토

### 표면 요청
"Inbox 탭과 Schedule 탭 중복되는 느낌이 많는데 합치고 기능 강화 및 편리성 강화 하는 방향으로 고려바람. 지금이 더 낫다면 지금 상태로 냅둬도 됨."

### 비판적 정리
- **Inbox**: 빠른 캡처(텍스트·음성), 한 줄 파싱 → 일정/노트, "일정으로 구조화하고 캘린더에 추가" 원탭 플로우.
- **Schedule**: 멀티턴 AI 대화로 주간 계획 수립 → 활동 목록 생성 → 요일/시간 조정 → "캘린더에 등록".
- **겹치는 부분**: 둘 다 "일정 만들기"와 "캘린더에 추가"로 이어짐. 사용자 입장에선 "뭘 써야 하지?"가 될 수 있음.
- **다른 부분**: Inbox = **캡처·파싱** 중심(짧은 입력 → 구조화). Schedule = **대화·설계** 중심(질문답하며 주간 배분).

### 접근 A: 유지 (역할 분리)
- **기준**: "빠르게 적기" vs "계획 세우기"가 명확하면 탭 2개가 오히려 인지 부하를 줄일 수 있음.
- **조건**: 앱 내에서 두 탭의 **진입 문구·온보딩**이 "Inbox = 메모/할 일 적기", "Schedule = 주간 일정 설계"로 구분되어 있을 때 유지 권장.

### 접근 B: 통합 (단일 진입 + 모드)
- **내용**: 하나의 탭(예: "Plan" 또는 "Inbox") 안에서 **모드 전환**: "빠른 기록" vs "AI와 주간 설계". 상단에 세그먼트 또는 탭으로 전환. 기록 모드에서는 현재 Inbox처럼 입력·파싱·캘린더 추가; 설계 모드에서는 현재 Schedule처럼 채팅·일정 생성·캘린더 등록.
- **장점**: "일정 만드는 곳"이 한 곳으로 모임. 탭 수 감소(6→5).
- **단점**: 한 화면 안에 두 플로우가 공존해 UI 복잡도 증가. 기존 사용자 습관 변경.

### 권장
- **비교 기준**: (1) 신규 사용자 인지 부하, (2) "일정 추가"까지의 단계 수, (3) 유지보수 비용(두 탭 vs 한 탭 두 모드).
- **현 단계 권장**: **유지(접근 A)**. 이유: (1) 역할이 다름(캡처 vs 설계). (2) 통합 시 라우팅·상태·AI 컨텍스트 분리가 커짐. (3) "중복"을 줄이려면 **진입점 안내 강화**(예: Inbox 부제 "빠른 기록 · 한 줄로 일정 만들기", Schedule 부제 "AI와 주간 설계")로 해소 가능.  
- **통합을 검토할 시점**: 사용자 피드백으로 "Inbox와 Schedule 중 뭘 써야 할지 모르겠다"가 반복될 때, 별도 Plan/Design으로 "Inbox+Schedule 통합" 기능을 두고 진행.

## 3. UI/UX가 목업 수준에 머무는 이유 — 규칙 vs 기술 역량

### 표면 요청
"UI/UX가 목업 수준에서 머무르는 이유가 강력한 상위 규칙에 묶여 있기 때문인지 기술 역량 부족인지 확인 후 보고바람."

### 확인 내용

#### 3.1 상위 규칙(.cursorrules 등)의 영향
- **규칙 요약**: Design-First(설계 문서 없이 기능 구현 금지), 타입/검증, AI 표준, Mobile-First UX(터치 타겟, 로딩/에러/빈 상태).
- **목업 수준과의 관계**:  
  - **규칙이 직접 "목업처럼 하라"고 하진 않음.** 다만 **Design-First**는 "UI를 먼저 다듬고 나서 설계 문서 쓰기"보다 "설계 문서 → 구현" 순서를 요구하므로, **반복적인 비주얼 폴리싱**이 설계 문서 업데이트 없이 이뤄지기 어렵다.  
  - **결론**: 상위 규칙이 **과정(설계 우선)**으로는 영향을 주고, "목업 수준"을 **목표**로 고정한 것은 아님. 다만 설계 문서가 **시각/인터랙션 디테일**까지 담지 않으면, 구현이 "기능 중심"에 머물기 쉬움.

#### 3.2 기술적 요인
- **스택**: React Native + Expo + NativeWind(Tailwind 계열).  
- **가능 요인**: (1) **Modal + KeyboardAvoidingView + ScrollView** 조합에서 기기/OS별 레이아웃 차이로 겹침·압축 발생. (2) NativeWind가 일부 컴포넌트(Modal 내부, 조건부 스타일)에서 예상대로 적용되지 않는 사례 가능. (3) **디자인 토큰**(design-system.ts, layoutConstants)은 있으나, **컴포넌트별 사용 강제**가 없어 화면마다 하드코딩된 값이 남을 수 있음.  
- **결론**: "기술 역량 부족"이라기보다 **RN/NativeWind의 특성**(모달·키보드·폰트 스케일)과 **디자인 토큰 적용 일관성** 부족이 맞물려, 목업처럼 보이려는 의도가 구현에서 충분히 반영되지 않을 수 있음.

#### 3.3 종합 보고
- **상위 규칙**: 목업 수준을 **유지하라**고 한 것이 아니라, **설계 우선** 프로세스가 "빠른 비주얼 반복"을 줄이는 **간접적** 영향은 있음. 설계 문서에 **UI 스펙(여백, 폰트, 모달 크기)**을 구체적으로 쓰면 규칙과 충돌 없이 폴리싱 가능.
- **기술**: 플랫폼/라이브러리 특성(모달·키보드)과 토큰 일관성 부족이 **기술적 원인**에 가깝다. 역량보다는 **환경 설정·패턴 정리**로 상당 부분 해소 가능.
- **권장**: (1) 설계 문서에 **UI 상세 스펙**(모달 크기, 폰트 크기, 라인 높이, 터치 영역)을 포함하도록 템플릿 보강. (2) 캘린더 "일정 추가"는 **접근 B(큰 박스)** 스펙을 design에 반영한 뒤 구현. (3) 공통 폼·모달용 **레이아웃 유틸** 또는 컴포넌트를 두어, 모달 내부에서만 쓰는 패딩/간격을 토큰으로 고정.
