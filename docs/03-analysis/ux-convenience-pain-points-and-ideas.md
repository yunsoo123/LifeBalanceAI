# UX 불편 요인 파악 및 편리성 증진 방안

**Date**: 2026-02-22  
**Project**: Mendly (LifeBalanceAI)  
**목적**: 현재 앱 사용 시 유저가 느낄 수 있는 불편 요소를 정리하고, 편리성 증진 방안을 제안한다.

---

## 1. 해석 (의도)

| 표면 | 실제 의도 |
|------|-----------|
| 불편함 파악 + 편리성 방안 모색 | **발견 가능성(Discoverability)**, **일관된 언어/피드백**, **대기/오류 체감**, **반복 작업 감소** 관점에서 개선 포인트를 구체화하고, 우선 적용할 수 있는 액션을 정리한다. |

---

## 2. 전제

- 실제 유저 인터뷰/로그는 없음. 코드·화면 플로우·디자인 문서 기반으로 **추정**한 불편 요인이다.
- 개선안은 “바로 구현”이 아니라 **후보**로 두고, PDCA로 Plan → Design → Do 할 수 있도록 구체적으로 적는다.

---

## 3. 불편 요인 및 편리성 증진 방안

### 3.1 발견 가능성(Discoverability)

| 불편 요인 | 현재 상태 | 편리성 방안 |
|-----------|-----------|-------------|
| **할 일 삭제가 길게 누르기만 가능** | 롱프레스로 삭제 확인창 표시. 화면에 안내 문구 없음. | (1) 할 일 리스트 상단 또는 빈 상태에 "길게 누르면 삭제" 한 줄 안내 추가. (2) 또는 행 옆 [⋯] 메뉴에서 "삭제" 제공해 롱프레스와 병행. |
| **Capture에서 "일정만/할 일만/둘 다"가 첫 사용에 낯섦** | 저장 버튼 누르면 모달로 3가지 선택. 의미가 한눈에 안 들어올 수 있음. | (1) 모달 제목/설명에 "어디에 추가할까요?" + 짧은 설명(일정=캘린더, 할 일=체크리스트). (2) 첫 실행 시 툴팁 또는 온보딩 1장 추가. |
| **Calendar 탭에 "일정 | 할 일"이 있는지 진입 전에 모름** | 탭 이름이 "Calendar"라서 할 일이 여기 있다는 인지가 어려울 수 있음. | (1) 가이드/온보딩에서 "캘린더 탭에서 일정과 할 일을 함께 볼 수 있어요" 안내. (2) Capture 빈 상태 CTA에 "할 일 보기 → Calendar" 링크. |

### 3.2 언어·메시지 일관성 (i18n)

| 불편 요인 | 현재 상태 | 편리성 방안 |
|-----------|-----------|-------------|
| **Alert/한도 메시지가 locale과 무관하게 한글만** | 예: `무료 플랜은 월 N회까지...` 가 lang=en일 때도 한글. | 한도·로그인 필요·실패 메시지를 전부 i18n 키로 빼고, locale에 따라 한/영 문구 표시. |
| **온보딩 슬라이드가 title/subtitle은 영문, description은 한글** | 슬라이드가 혼용. | locale에 따라 title/subtitle/description 모두 한글 또는 모두 영문으로 통일. |
| **오프라인 배너가 영문 고정** | "Offline — changes will sync when connected". | i18n 추가 (예: common.offlineMessage), 한/영 전환. |
| **sign-in 푸터 "계정이 없으신가요?"** | 일부 화면에서 하드코딩 한글. | t.auth.createAccountPrompt 등 i18n 키로 통일. |

### 3.3 로딩·피드백 체감

| 불편 요인 | 현재 상태 | 편리성 방안 |
|-----------|-----------|-------------|
| **목록 새로고침이 수동만 가능** | Calendar(일정/할 일), Notes, Review에 **pull-to-refresh 없음**. | ScrollView/FlatList에 RefreshControl 추가해 당겨서 새로고침 가능하게 함. |
| **Capture 저장 중 버튼만 스피너** | 배경은 그대로. 네트워크 느릴 때 "진행 중인지" 불명확할 수 있음. | (1) 저장 중에는 입력창/버튼 비활성화 + 스피너 유지. (2) 3초 이상 걸리면 "잠시만 기다려 주세요" 토스트(선택). |
| **API 실패 시 메시지가 기술적** | 일부에서 `HTTP 500` 또는 raw error.message 노출 가능성. | 네트워크/서버 에러는 "연결을 확인해 주세요" / "잠시 후 다시 시도해 주세요" 등으로 치환해 사용자 문구만 노출. |

### 3.4 반복 작업·효율

| 불편 요인 | 현재 상태 | 편리성 방안 |
|-----------|-----------|-------------|
| **할 일 추가 후 매번 "할 일 추가" 영역 닫기** | Calendar 할 일에서 인라인 입력 후 추가하면 폼이 닫힘. 반복 추가하려면 다시 "+ 할 일 추가" 탭. | (1) 추가 후 입력란만 비우고 폼은 유지하는 옵션. (2) 또는 "추가 후 하나 더 추가" 체크 시 폼 유지. |
| **Capture에서 "일정만" 선택 시 Parse 실패 시 재시도가 번거로움** | Alert만 띄우고 입력 내용이 그대로면 사용자가 다시 저장 버튼 누름. | 실패 Alert에 [다시 시도] 버튼 제공. (이미 입력된 텍스트로 재요청.) |
| **Calendar 일정 추가가 여러 필드** | 제목, 시작/종료 시간, 설명 등. 빠르게 "제목만" 넣고 싶을 때 단축 플로우 없음. | (선택) "빠른 추가": 제목만 입력하면 오늘 09:00–10:00 기본값으로 일정 생성. 상세는 편집에서. |

### 3.5 오프라인·에러 경계

| 불편 요인 | 현재 상태 | 편리성 방안 |
|-----------|-----------|-------------|
| **오프라인 시 저장 실패 메시지가 일반 "저장 실패"** | 네트워크 끊김과 서버 에러가 구분되지 않을 수 있음. | useNetStatus() 사용해 오프라인일 때 "오프라인이라 저장되지 않았어요. 연결 후 다시 시도해 주세요." 등 명시 메시지. |
| **한도 초과 시 "Pro 업그레이드"만 안내** | 인앱 결제 미구현이면 사용자가 다음 액션을 모름. | "이번 달 한도를 다 썼어요. 다음 달에 다시 쓸 수 있어요." + (선택) "Pro는 준비 중이에요" 등으로 기대치 조정. |

### 3.6 접근성·터치 영역

| 불편 요인 | 현재 상태 | 편리성 방안 |
|-----------|-----------|-------------|
| **일부 버튼/링크에 accessibilityLabel 없음** | 스크린 리더 사용 시 의미 전달이 부족할 수 있음. | Capture 저장 버튼, 배치 선택 옵션, Calendar 세그먼트, 할 일 행 등에 accessibilityLabel(및 필요 시 accessibilityHint) 추가. |
| **할 일 행 터치 영역** | 탭=완료 토글, 롱프레스=삭제. 영역은 48pt 이상이면 무난. | 롱프레스에 accessibilityHint="길게 누르면 삭제 메뉴" 등으로 보조 설명. |

## 4. 우선순위 제안

| 우선순위 | 항목 | 이유 | 예상 공수 |
|----------|------|------|-----------|
| **P0** | 한도/로그인/오프라인 등 **Alert·메시지 i18n 통일** | 언어 설정과 무관하게 한글이 나오면 영문 사용자에게 불편. | 소 |
| **P0** | **Pull-to-refresh** (Calendar 일정·할 일, Notes, Review) | 목록이 갱신되지 않을 때 당기는 습관이 있어 기대에 부합. | 소 |
| **P1** | 할 일 **삭제 안내** (길게 누르기 또는 [⋯] 메뉴) | 삭제 가능함을 모르면 불편. | 소 |
| **P1** | **오프라인 배너·저장 실패 시 오프라인 메시지** i18n + 네트워크 구분 | 오프라인 체감 명확화. | 소 |
| **P2** | 온보딩 **locale별 title/description 통일** | 첫인상 일관성. | 소 |
| **P2** | Capture **저장 실패 시 [다시 시도] 버튼** | 재시도 편의. | 소 |
| **P3** | 할 일 **추가 후 폼 유지** 옵션, **빠른 일정 추가** 등 | 반복 작업 효율. | 중 |

## 5. 접근 방식 정리

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **A) 바로 Do** | 위 P0·P1부터 순서대로 코드 수정. | 빠르게 체감 개선. | 범위가 넓어 누락 가능. |
| **B) Plan → Design → Do** | "UX 편의성 개선"을 하나의 feature로 Plan/Design 문서 작성 후, 항목별 체크리스트로 Do. | 추적·검증 용이. | 문서 작성 시간 필요. |

**권장**: **P0 두 가지(i18n 통일, Pull-to-refresh)** 는 범위가 명확하므로 별도 Design 없이 **Minor change**로 진행해도 됨. P1 이후는 원하면 `docs/01-plan/features/ux-convenience-improvements.plan.md` 를 만들어 체크리스트로 관리.
