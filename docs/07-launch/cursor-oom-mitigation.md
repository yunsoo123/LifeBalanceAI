# Cursor OOM(Out Of Memory) 대응

**Date**: 2026-02-22  
**Project**: Mendly (LifeBalanceAI)

---

## 1. 현상

- Cursor IDE 창이 **reason: 'oom', code: '-536870904'** 로 종료됨.
- 동일 작업(Phase B 등) 진행 중 반복 발생 가능.

---

## 2. 원인 분석

| 가능 원인 | 설명 |
|-----------|------|
| **대용량 컨텍스트** | 한 세션에서 많은 파일 열기·검색·대용량 문서 로드 시 메모리 사용량 증가. |
| **대형 파일** | `app/(tabs)/calendar/index.tsx` 등 3000줄 이상 파일을 여러 개 열어두면 비용 증가. |
| **동시 도구 사용** | tsc, lint, 여러 grep/read를 동시에 실행하면 일시적으로 메모리 피크. |
| **에이전트 컨텍스트** | PDCA로 plan/design/do를 한 번에 진행하면 대화·파일 내용이 길어져 OOM 위험 증가. |

---

## 3. 개선 방안 (적용·권장)

### 3.1 작업 단위 분할 (권장)

- **한 번에 하나의 feature/항목만** 진행.  
  예: Phase B에서 2 완료 → (커밋/저장) → 6 → (커밋) → 7.
- design·do를 나눠서 진행하고, do 시 해당 feature 파일만 집중 편집.

### 3.2 불필요한 파일 제외 (.cursorignore)

- `node_modules/`, `dist/`, `.expo/`, 빌드 산출물을 이미 제외했는지 확인.
- 대용량 로그·캐시 폴더가 인덱싱되지 않도록 유지.

### 3.3 에디터 사용 습관

- **필요한 파일만** 열기. 작업 끝난 탭은 닫기.
- 매우 큰 파일은 **특정 구간만** 검색/읽기로 참조하고 전체를 열지 않기.

### 3.4 Cursor/VS Code 설정

- **설정** → "memory" 검색 → `window.restoreWindows` 등 복원 옵션 확인.  
  재시작 시 모든 창/탭 복원을 끄면 다음 실행 시 메모리 부담 감소.
- **Don't restore editors** 체크 후 Reopen 시 이전 편집기 복원을 줄여 메모리 절약.

---

## 4. 진행 복구 시 권장 순서

1. Reopen 후 **진행 상황만** 확인 (status, 완료된 항목 목록).
2. **남은 항목 1개씩**만 진행 (예: 6 완료 → 저장 → 7만 진행).
3. 각 항목 완료 후 **tsc·lint**만 실행하고, 필요 시 커밋 후 다음 항목 진행.

---

**참고**: OOM은 Cursor/Electron 프로세스 한계로 발생하므로, 코드 변경만으로는 해결되지 않음. 작업 단위 축소와 리소스 사용 최소화가 핵심.
